# 基于微服务架构的分布式音乐流媒体服务平台项目说明

更新时间：2026-04-08

## 1. 项目概述

本项目面向“基于微服务架构的分布式音乐流媒体服务平台”这一毕业设计课题，目标是构建一套围绕音频播放、视频播放、用户资产管理、推荐服务、后台管理和 Web 端展示的完整服务端系统。项目以 Go 语言为主要后端实现语言，采用 Nginx 作为统一入口，通过 Docker Compose 完成容器化编排，并配套 Redis、MySQL、OpenAPI 文档、后台管理页面以及 Web 前端播放器，形成一个可部署、可验证、可扩展的工程化平台原型。

选择音乐流媒体平台作为课题载体，主要有三个原因。第一，音乐与视频流媒体业务天然同时包含静态资源分发、动态接口处理、用户状态维护、媒体元数据管理与后台治理等典型服务端问题，具有较强的综合性。第二，该类系统既包含同步请求链路，也包含异步事件链路，适合用来承载微服务拆分、网关聚合、事件可靠性、推荐反馈等分布式设计议题。第三，相比纯理论性的系统题目，流媒体平台更容易形成可演示、可测试、可扩展的完整成果，便于毕业设计阶段展示系统设计与工程落地能力。

需要说明的是，本项目当前更适合被定义为“面向分布式部署的微服务平台原型”，而不是面向海量生产流量的超大规模分布式集群。项目已经具备分布式系统的核心特征，例如服务拆分、网络协作、异步事件、独立容器部署和静态资源分层，但尚未进一步引入 Kubernetes、多副本自动调度、服务注册发现、跨机房高可用等更大规模的基础设施能力。

## 2. 项目演进背景

本项目并非从零开始一次性搭建完成，而是在原有部署目录基础上逐步演进形成。早期系统主要面向单机部署，业务能力集中在一个服务进程中，静态音频与视频资源依赖本地目录直接提供，配置方式也偏向手工维护。这种方式虽然可以满足基础播放和简单接口调用，但在系统扩展、职责划分、部署迁移和功能演进方面存在明显局限。

随着项目功能增多，系统逐步开始围绕领域职责进行拆分，形成了认证、目录、用户画像、媒体、视频和事件处理等多个服务。与此同时，项目开始引入 Nginx 网关统一暴露外部接口，并将音频、视频、歌词、HLS 文件等静态资源交由 Nginx 或媒体服务承担，使业务接口处理和大文件访问分离，从而减少 Go 应用层在文件 I/O 上的压力。

在工程落地层面，系统又从“本地脚本 + 手工配置”的启动方式，继续演进为 Docker 一键部署模式。项目通过 `docker-compose.yml` 同时编排 MySQL、Redis、Gateway、Auth Service、Catalog Service、Profile Service、Media Service、Video Service、Event Worker 等多个容器，并通过 `.env.docker` 驱动运行参数。后续又引入“双目录工作流”，将 `microservice-deploy` 固定为开发测试仓库，将 `CloudMusic` 固定为正式运行目录，从而形成一套更清晰的开发、验证、推送和正式部署流程。

从演进视角看，本项目的核心变化可以概括为四条主线：

1. 从单体式后端向领域拆分微服务演进。
2. 从应用直出媒体资源向网关聚合与静态资源分离演进。
3. 从同步接口为主向同步接口与异步事件并存演进。
4. 从手工部署向容器化、一键部署和正式发布流程演进。

## 3. 总体架构设计

### 3.1 总体分层

当前系统可以抽象为五层结构：

1. **网关层**
   由 Nginx 统一对外暴露 `8080` 端口，对客户端请求进行入口聚合、路由转发和静态资源分发。网关同时承担用户接口、推荐接口、歌单接口、管理后台接口、静态音视频访问入口等统一暴露职责。

2. **业务服务层**
   按领域拆分为多个 Go 服务进程：
   - `auth-service`：用户注册、登录、在线会话、用户资料。
   - `catalog-service`：音乐目录、搜索、歌手检索、基础音频列表。
   - `profile-service`：喜欢、播放历史、私有歌单、推荐相关接口。
   - `media-service`：上传、下载、歌词、媒体文件服务。
   - `video-service`：视频列表和视频流地址生成。
   - `event-worker`：消费领域事件、处理异步任务和可靠事件链路。

3. **静态资源层**
   音频文件、视频文件和 HLS 切片目录由宿主机路径挂载到容器内，对外通过 `/uploads/`、`/video/` 等路由直接提供访问。歌词请求中涉及映射关系的部分由媒体服务进行辅助处理。

4. **数据存储层**
   主要使用 MySQL 和 Redis。MySQL 承担业务数据、推荐数据、事件 outbox 数据等持久化职责；Redis 既承担缓存，也承担在线状态和 Redis Streams 事件总线能力。

5. **文档与运维层**
   项目配备 OpenAPI 文档、Swagger UI、本地部署脚本、Docker 部署脚本、媒体目录入库脚本以及正式部署工作流文档，为测试、联调和后续维护提供支持。

### 3.2 服务职责边界

当前微服务职责划分较为明确：

- `auth-service` 面向账号与会话相关能力，负责注册、登录、在线心跳、资料查询、用户名修改、头像上传等与“身份”和“会话”强相关的接口。
- `catalog-service` 面向内容目录，负责歌曲列表、搜索、歌手维度过滤、音乐基础元数据读取。
- `profile-service` 面向用户音乐资产和个性化能力，负责喜欢、播放历史、歌单和推荐接口，是用户行为与偏好相关的聚合服务。
- `media-service` 面向媒体文件处理与上传下载，负责上传歌曲、下载文件、歌词访问、媒体路径服务等能力。
- `video-service` 面向视频列表与视频流地址生成，承担独立的视频资源接口。
- `event-worker` 面向异步事件消费，配合 Redis Streams 和 outbox 实现事件可靠投递、重试和异步处理。

### 3.3 Monolith 与微服务并存关系

项目中同时存在 `cmd/monolith/main.go` 和多个微服务入口。Monolith 入口保留了兼容聚合能力，便于本地调试和功能回归；而 Docker 与分离部署模式下则主要通过 Nginx 将请求分发到各个领域服务。两者并存的设计，使项目既能够快速保持旧接口兼容，又能逐步向更清晰的微服务架构演进。

### 3.4 为什么该项目具有“分布式”属性

从工程事实出发，本项目与“分布式”相关的部分主要体现在以下几个方面：

1. 系统已经拆分为多个可独立运行的服务进程，各服务通过网络接口协同，而不是单体内部函数调用。
2. 外部访问统一经过网关层，由网关完成路由聚合与静态资源转发，体现了典型的分布式入口治理方式。
3. 系统引入了基于 Redis Streams 的异步事件链路，服务之间不只依赖同步调用，还可以通过事件实现解耦与最终一致性。
4. 各服务以独立容器部署，可在后续演进为多实例和多节点部署，而不是与单一进程强绑定。

因此，本项目在毕业设计语境下可以被准确描述为：具备微服务拆分、网络协作、异步事件与容器化独立部署能力的分布式流媒体平台原型。

## 4. 核心业务模块说明

### 4.1 用户体系

用户体系目前已经覆盖注册、登录、在线状态、用户资料和兼容收藏链路等多个方面。基础账号接口由 `auth-service` 暴露，登录后除了返回基础账户信息外，还会返回在线会话参数，用于后续心跳续约和在线状态维护。

为解决旧版接口容易被伪造、在线状态无法主动回收的问题，系统新增了在线会话机制，围绕 `session_token` 建立起创建、心跳、状态查询和主动下线一整套接口流程。该机制以 Redis 为主要存储介质，通过 TTL 和双向绑定校验强化了在线状态的可信度。

在用户资料方面，系统已支持查询当前资料、修改用户名和上传头像。头像文件通过 `multipart/form-data` 上传后落盘到受管目录，并对外返回可直接访问的 `avatar_url`。同时，为兼容旧版收藏链路，修改用户名时还会同步更新旧兼容表中的用户名字段，避免老接口失效。

### 4.2 音乐目录与检索

目录服务主要负责系统中“有哪些歌曲、如何找到这些歌曲”。系统已支持：

- 音乐列表获取
- 根据关键词搜索歌曲
- 按歌手过滤歌曲
- 歌手存在性搜索
- 音乐流地址生成

目录中的媒体元数据主要来自数据库表，而数据库中的媒体记录又可以通过本地目录扫描脚本自动导入。这样一来，系统既支持基于数据库的统一查询，也能适配本地静态资源目录作为实际媒体文件来源。

### 4.3 媒体资源服务

媒体资源模块覆盖文件上传、下载、歌词访问和静态文件服务。该模块有两个层次的职责：

1. 业务层：处理上传、下载、歌词映射、媒体元数据回填等逻辑。
2. 资源层：通过 Nginx 对音频、视频、HLS 切片等大文件进行直接访问。

这套设计的价值在于将“业务接口处理”和“大文件传输”解耦。动态逻辑继续由 Go 服务承担，而真正的文件访问由 Nginx 直出，从而降低后端进程在高频文件请求下的压力。

### 4.4 用户音乐资产

用户音乐资产包括喜欢列表、播放历史和私有歌单三部分。

喜欢与播放历史接口由 `profile-service` 统一管理，支持新增、删除、列表查询、清空等行为；同时这些用户行为还能进一步作为推荐系统的输入信号。私有歌单是后续新增的重要能力，支持歌单创建、修改、删除、详情查询、批量加歌、批量删歌和完整排序。歌单项同时兼容两类来源：

- 服务端已有的在线歌曲
- 客户端本地歌曲

这种设计增强了系统对不同播放场景的适配能力，也使客户端后续扩展“自建歌单”功能时能够复用统一数据模型。

### 4.5 推荐系统

推荐系统是本项目较有代表性的扩展模块之一，目前已经完成第一版落地。系统对外提供：

- 个性化推荐列表
- 相似歌曲推荐
- 推荐反馈上报
- 模型状态查询
- 手动触发重训任务

当前推荐策略为可运行的 `rule_hybrid` 规则混合模型，不依赖外部机器学习平台。该策略综合使用播放历史、喜欢列表、全局热度和用户反馈等信息完成加权排序，并返回推荐原因与来源字段，便于客户端展示和后续效果分析。

从毕业设计角度看，这套推荐实现的意义不在于算法复杂度本身，而在于证明平台已经具备“用户行为采集 -> 偏好建模 -> 推荐输出 -> 反馈闭环”的基本体系。

### 4.6 管理后台

项目内置了一套服务端后台管理页面，入口位于 `/admin`。管理员登录后可以完成以下操作：

- 查看用户列表
- 查看用户在线状态与最后在线时间
- 查看媒体资源
- 按歌手维度管理音频资源
- 批量删除数据库记录
- 批量删除数据库记录并同步删除物理文件
- 上传歌曲、歌词和封面
- 当封面缺失时使用 `ffmpeg` 自动提取

后台的存在使系统不仅能服务于客户端消费，也具备一定程度的运维和内容治理能力，增强了整体平台的完整性。

### 4.7 Web 端平台

除桌面客户端外，项目还配套实现了一个基于 React + TypeScript + Vite 的 Web 端音视频平台，位于 `web-player/`。该平台能够完成服务器连接、登录注册、音频播放、视频播放、推荐展示、搜索、播放队列、歌词展示、喜欢列表和行为反馈等功能。

Web 端的意义有两点。第一，它证明服务端接口具备跨终端复用能力，而不是只服务于单一桌面客户端。第二，它为后续展示系统“多端协同”的能力提供了更加直观的材料。

## 5. 关键技术与工程化实现

### 5.1 微服务拆分策略

项目采用按业务领域拆分的方式，而不是按技术层拆分。认证、目录、用户资产、媒体、视频和事件消费分别落在不同服务中，这种拆分方式有利于明确边界、降低耦合，也便于后续按服务独立扩缩容。

### 5.2 Nginx 网关统一入口

Nginx 网关承担统一入口角色，所有对外访问均可经由 `8080` 端口完成。网关根据不同路径将请求路由到对应服务，例如用户接口转发到 `auth-service`，用户资产和推荐接口转发到 `profile-service`，目录接口转发到 `catalog-service`，视频接口转发到 `video-service`，后台页面和部分兼容入口转发到 `music-server`。同时，音频、视频和 HLS 资源可以直接由 Nginx 对宿主机目录做静态映射。

这种设计一方面隐藏了微服务内部端口和部署细节，另一方面也为后续加入认证、限流、统一日志和统一错误处理提供了扩展入口。

### 5.3 静态资源分离

在流媒体场景中，音频、视频和切片文件通常体积较大、访问频率高，如果全部由应用层处理，会带来额外 I/O 压力。当前项目将静态资源访问从业务处理链路中剥离，由 Nginx 通过目录映射和 `alias` 直接对外提供访问，Go 服务则专注于元数据、上传和业务逻辑处理。

这一点是项目架构中的一个关键亮点，因为它体现了“动态请求”和“静态媒体分发”分层处理的思想，也是流媒体平台工程化的重要基础。

### 5.4 事件可靠性设计

项目在事件侧采用了 Outbox + Redis Streams + Consumer Group 的组合方案。其核心思路是：

1. 业务服务在本地操作成功后尝试发布领域事件。
2. 若直接发布失败，则将事件写入 `event_outbox` 表，避免事件丢失。
3. 后台轮询或工作线程从 outbox 表取出待投递事件并重新发布到 Redis Streams。
4. `event-worker` 作为消费者组成员消费事件、确认消息，并处理重试、积压恢复等场景。

这一机制的价值在于为跨服务的数据协同和后续扩展提供了更可靠的事件基础，而不是完全依赖同步接口串联。

### 5.5 在线状态体系

早期系统在线状态依赖较弱的兼容接口，存在被伪造和缺少主动回收的问题。当前项目已经围绕 `session_token` 构建起标准化在线状态体系，包括会话创建、心跳续约、状态查询和主动下线，并在 Redis 中维护会话 token 与账号的绑定关系。

这一改造既提升了在线状态的准确性，也使客户端在登录、切换账号、退出应用等场景下能够采用更规范的服务端交互方式。

### 5.6 Docker 一键部署

项目当前的主部署方式是 Docker 一键部署。`start_docker.sh` 会调用内部脚本完成环境文件读取、配置渲染、容器构建与服务启动。`.env.docker` 则承担部署参数入口角色，用于配置：

- Compose 项目名
- 对外端口
- MySQL/Redis 密码
- 公网访问地址
- 音频目录、视频目录、HLS 目录
- 管理员账号
- 领域 schema 名称

这一设计将部署环境差异从代码层抽离出来，有利于代码迁移和环境切换。

### 5.7 双目录开发与正式部署工作流

为避免开发目录和正式运行目录混用，项目后续又建立了双目录工作流：

- `microservice-deploy`：唯一开发仓库，用于编码、测试、临时 Docker 验证。
- `CloudMusic`：唯一正式运行目录，固定跟踪 `origin/main`。

正式部署由 `CloudMusic` 中的 `deploy_from_main.sh` 负责，该脚本会先验证当前分支和工作区状态，再拉取最新 `main`，调用 `start_docker.sh` 重建正式容器，并执行健康检查。该工作流的建立，使项目在毕业设计阶段已经具备了比较清晰的“开发 -> 推送 -> 正式更新”流程。

### 5.8 文档体系建设

项目配备了较为完整的文档体系，包括：

- README 与部署说明
- OpenAPI 文档与 Swagger UI
- 在线状态说明
- 推荐接口说明
- 私有歌单说明
- 用户资料说明
- Web 端说明
- 双目录部署流程说明

文档体系的存在使项目不仅能被运行，也能被他人理解、复现和继续开发，这对毕业设计项目的可维护性和可展示性具有实际价值。

## 6. 数据库与数据域设计

### 6.1 数据域拆分思路

当前系统并未发展到严格的分库分片阶段，但已经通过 schema 形式体现出数据域拆分思路。配置中主要涉及三个领域 schema：

- `music_users`：目录与用户侧基础信息
- `music_profile`：用户资产、推荐相关能力
- `music_media`：媒体文件和歌词映射等能力

此外，事件可靠性相关表也通过专门迁移落库，用于支撑 outbox 机制。

这种数据域划分的价值在于为服务边界提供更清晰的数据归属基础，避免所有能力混杂在同一张库表中。

### 6.2 重点数据能力

从当前实现来看，数据库主要承载以下数据能力：

1. **目录数据**
   包括歌曲路径、标题、歌手、专辑、时长、歌词路径、封面路径等元数据。

2. **用户资产数据**
   包括喜欢、播放历史、私有歌单及其曲目项。

3. **推荐数据**
   包括推荐反馈、模型状态和训练任务记录。

4. **事件可靠性数据**
   包括待投递事件、发布状态、重试信息等 outbox 数据。

5. **用户资料数据**
   包括用户名、账号、头像路径以及与兼容链路相关的信息。

### 6.3 媒体目录扫描与写库机制

为了适配“资源文件在本地目录、业务逻辑依赖数据库元数据”的场景，项目提供了媒体扫描入库脚本。该脚本可以扫描指定音频和视频目录，读取标签、时长、歌词路径和封面信息，并将结果幂等写入数据库。Docker 镜像中已内置 `ffmpeg/ffprobe`，从而支持在容器环境中完成媒体信息提取。

这一机制解决了“代码和接口已经就绪，但替换了资源目录后数据库中没有元数据”的问题，是平台在部署迁移时非常关键的一步。

## 7. 当前成果与系统特征总结

经过多轮迭代，当前项目已经形成了较为完整的流媒体平台能力闭环，主要体现在以下几个方面：

1. 已完成从单体式服务向领域微服务的基本拆分。
2. 已形成 Nginx 网关统一入口与静态资源分离架构。
3. 已支持 Docker 一键部署与正式部署流程约束。
4. 已具备推荐、歌单、用户资料、在线状态等较完整的用户侧能力。
5. 已具备管理后台，能够支撑内容治理和用户状态查看。
6. 已具备 Web 端平台，证明后端接口可被多端复用。
7. 已配套 OpenAPI 与专项文档体系，具备较好的工程可维护性。

如果从论文表述角度总结项目特色，可以概括为：

- **微服务分布式架构**：按领域拆分服务，通过网关聚合外部访问，实现服务解耦与独立部署。
- **静态资源分离架构**：将音频、视频和 HLS 资源交由独立资源层处理，减轻业务服务 I/O 压力。
- **事件可靠性架构**：基于 Outbox 与 Redis Streams 设计异步事件链路，提高跨服务协同的可靠性。
- **工程化交付能力**：通过 Docker、配置驱动、部署脚本和双目录流程提升部署和维护效率。
- **能力可扩展性**：在原有播放与目录基础上，继续演进出推荐、歌单、资料、后台和 Web 端能力。

同时也应当清楚指出，本项目当前仍属于工程化分布式原型。它已经具备可运行、可部署、可扩展和可展示的完整平台雏形，但并不以“大规模生产级集群”作为当前阶段目标。

## 8. 局限性与后续优化方向

虽然当前项目已经具备较丰富的能力，但从系统架构和工程成熟度角度看，仍存在一些可以继续优化的方面。

第一，项目当前主要依赖单机 Docker Compose 进行容器编排，尚未进一步升级为 Kubernetes 或其他多节点调度平台，因此在多副本自动调度、节点级高可用、弹性扩缩容方面仍有提升空间。

第二，虽然服务已经完成基础拆分，但部分业务仍集中在 `profile-service` 中，例如喜欢、历史、歌单和推荐仍处于同一服务边界内。后续若继续深化微服务架构，可以考虑进一步将推荐或用户资产拆为更独立的服务。

第三，项目尚未引入完整的服务注册发现、分布式链路追踪和系统级指标采集能力。当前已有健康检查、日志和基本脚本，但若面向更高复杂度部署，仍需要补充可观测性体系。

第四，静态资源目前主要以本地目录挂载方式提供，虽然已能满足毕业设计阶段的部署需求，但未来若面向更强的可扩展性，可以继续向对象存储、CDN 或 MinIO 方案演进。

第五，推荐系统当前采用规则混合策略，强调可运行和闭环验证，而不是高复杂度模型算法。后续若有更多时间和数据基础，可以继续引入更完整的内容特征建模或协同过滤优化。

综合来看，项目后续较合理的演进方向包括：

1. 将推荐能力进一步独立为单独服务。
2. 增强服务级日志、指标与链路追踪。
3. 引入多实例部署和更标准的调度平台。
4. 将媒体资源存储从本地目录升级为对象存储。
5. 继续完善推荐模型、后台治理和多端协同能力。
